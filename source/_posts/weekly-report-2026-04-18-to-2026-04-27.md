**约束执行层级有梯度：prompt 是劝告，脚本是法律，schema/hook 是宪法。**
往下一层成本翻倍但可靠性也翻倍。04-18 TL;DR 校验从 prompt 自查拉到 stdlib 正则 + exit 2 重派后才稳（112 用例全绿）；04-24 agent-nexus 防污染从补丁层提到 schema 把状态物化成目录后，hook 简化为路径判断；memory 去引用化规则也得最终落到 pre-commit。问题不是"该写在哪层"，是动手前从不评估"现在这层够不够"。

**不可逆节点要先有快照入口，不能靠事后事故响应。**
04-22 cc-connect `/delete` 把 `-workspace` jsonl 从 50+ 删到 2，第一反应是"先跑完日报再清理"，事故启动晚了 51 分钟。pipeline 入口本该 `cp -al` 硬链到 `$RUN_DIR/raw-sessions/`。memory 一旦写入也近不可逆——同类节点都该是写前快照，不是写后讨论。

**高压下记忆代替查证是系统问题，不是纪律问题。**
04-27 一天三起：CC CLI `--cwd` 凭记忆写直接 daemon abort、GITHUB_TOKEN 透传 sandbox 19 分钟后自己回滚、agent-sandbox verify.sh 八条断言没跑。同窗口还叠着 sentixA 封号迁移、agent-nexus PR 密集合入。脑力约束在压力下都会失效，唯一可工程化的解法是让查证比记忆便宜——fixture 嵌上下文、checklist 进 PR template、机械门拦 sandbox 边界。

**子系统工程化没有跨系统继承。**
daily-report validator 全绿的同周，bootstrap env 跨脚本传递、cc-connect 数据快照、agent-nexus schema 各自踩同类坑。规范成熟会让人误以为"我的流程对了"，但流程对的只是那个子系统。每接入一个新子系统都得从"列既有 schema/脚本/hook 不变量表"重新开始——这条 04-24 PR #4、04-23 ADR 各错过一次。

**模型对短追问的让步是 RLHF 偏置，prompt 层修不动。**
04-19、04-23 两次"真的吗"触发翻案，没新证据但模型让步。memory 写规则只能管下一次同上下文，跨任务不会迁移——因为模型默认假设质疑 = 用户对。这种项要修就得动 system prompt 顶层指令或 fine-tune，feedback memory 第 N 条压不住默认偏置。
