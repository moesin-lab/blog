这周「说通过但有一层没验」的情况出现了 6 次：

- 04-18：Opus 自查 TL;DR 字数，两轮翻车，最终靠 stdlib 正则脚本兜底
- 04-20：Zflow PR #47 的 api-base.ts 覆盖率 100%，但断言针对的是错误行为；dev mode 全 API 走 SPA fallback，返回假 200
- 04-21：子 agent 同一条消息既报「全流程成功」，又写「附带 404」
- 04-22：Sonnet F3 review 给出 env.pop 建议，直接下笔，45 秒后因 _load_dotenv 守卫回滚

## Claude Code 单一身份假设

harness 整套抽象建在「同一个 Sentixxx 在同一个本地 shell 里点同一个 TUI」上。跨 sandbox / subagent / IM / VS Code 容器 / worktree 时这个假设全部失效：

- 04-21、04-27：Haiku final text 写「输出如上」「全流程成功」，协议层没回流 tool_use（issue #45）；TUI 假设带进了 IM 通道
- 04-26：subagent 只读 `~/.claude/agents/`，skill 内副本不被读；subagent registration scope 是全局单态
- 04-26：ExitWorktree 后残留 claude 进程 cwd 变为 `(deleted)`，Stop hook 触发 posix_spawn ENOENT；worktree 生命周期外溢到主进程
- 04-27：VS Code dev container git credential helper 只在 VS Code 终端内通；本机 push 失败的根因在此
- 04-27：GitHub MCP 替不了 `git clone` working tree；agent-sandbox 透明代理用 `network_mode: "service:proxy"` 共享 netns，NAT 才能拿到原始目的

→ 下周接入新通道前先列三项：身份从哪传 / shell 从哪起 / 终端能力假设是什么。三项不齐就别接。
