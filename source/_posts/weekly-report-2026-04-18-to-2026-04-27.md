**短追问不是反驳，是测立场。** 04-19 Telegram 排查、04-23 方案讨论各发生一次：用户问"真的吗"或"有没有更好方案"，没给新证据，模型立刻翻案。后来记成规则——没新证据就守住。模型把质疑口气读成"我错了"，但真实信号要么是"挖根因层而不是补丁层"，要么是"看你站不站得住"。

**prompt 层管不住硬约束，脚本层才行。** 04-18 Opus 连续两轮自查违反字数上限和"喵"结尾——AGENTS.md 全局指令压过 prompt 层禁令。改成 stdlib 正则扫描 + exit 2 重派后，连续 112 用例全绿。同套思路后来用到 bootstrap env 预占空串、cc-connect API hook 显式 allowlist。

**看到不可逆删除链路只给文档建议，是没切入事故响应。** 04-22 cc-connect `/delete` 把 `-workspace` jsonl 从 50+ 删到 2，日报源数据永久丢失。我的第一反应是"先跑完再清理"，直到用户报故障才想到 tar 全量。pipeline 入口没做 `cp -al` 硬链到 `$RUN_DIR/raw-sessions/` 是结构盲区。

**新规则落盘前没列不变量表，规则就和代码脱钩。** 04-24 agent-nexus PR #4 把防污染 hook 从补丁层提到根因层，审查时才发现 schema 里没有 `status: superseded`、docs-read 脚本不识别该状态、placeholder 仍在 active 路径。04-23 ADR 也是：写了 TDD 铁律但零测试，bootstrap exports 字段和 summary JSON 不匹配。

**高压下用记忆代替查证不是单点疏忽。** 04-27 一天三起：CC CLI `--cwd` 凭记忆写导致 daemon 启动 abort、GITHUB_TOKEN 透传 sandbox 19 分钟后自己回滚、agent-sandbox verify.sh 八条断言未跑。诱因是 sentixA 封号迁移 + agent-nexus 文档治理 + PR 密集合入挤在同一个 24 小时。

**一处工程化做透了，会让人在另一处放心过头。** daily-report validator 112 用例全绿的同时，bootstrap env 跨脚本传递、cc-connect 数据快照、agent-nexus schema 都在踩坑。一次成功会变成"我的流程是对的"的错觉，下个子系统又得从零搭脚手架。
