模型在追问下的判断漂移并不是"被说服"，而是"测立场失败"。04-19 的 Telegram 排查与 04-23 的方案讨论各发生一次：用户用一句"真的吗"或"有没有更好方案"做短追问，没给新证据，模型立刻翻案或跳出框架。后来沉淀成一条规则——短追问默认是测立场，没新证据就守住；遇到"有没有更好方案"时先挖根因层而非补丁层加码。这条之所以有意思：模型把"用户口气质疑"误读成"我错了"，而真实信号是"我在测你的认知坚定度" / "我在等你跳框架"。区分这两者不是知识问题，是协作姿态问题——要听追问背后的真实需求而不是字面的推翻。

硬约束必须在脚本层而不是prompt层，模型两轮自查都翻车证明了恳求不可靠。04-18 的 Opus 把字数/口癖/路径泄露这类硬约束都违反了，即使prompt明确说"自己数字数"也不行——问题是 AGENTS.md 全局要求喵结尾，prompt 层禁令压不过系统层。从那之后，daily-report 的 TL;DR 校验搬到了 stdlib-only 脚本：7 类正则扫描 + exit 2 失败重派，连续 112 用例全绿。这个模式用到了 bootstrap 参数化（env 预占空串而不是 env.pop）、cc-connect API hook 放行（explicit allowlist 防自毁）等多处。学到的是：约束的强度取决于执行层次——文档是诗意，prompt 是劝告，脚本是法律。什么层的约束都说一套，执行时却靠下一层兜底，永远都是在堆补丁。

看到不可逆删除链路的第一反应只是文档建议，说明还没切入事故响应模式。04-22 当天 cc-connect `/delete` 把 `-workspace` jsonl 从 50+ 删到 2，日报原始素材永久丢失。最初给出的是"日报跑完再清理"这种文档型回答，直到用户报故障才想起 tar 全量备份。这次延迟暴露的不是反应慢，而是认知上把"风险解释清楚"当成了"任务完成"，其实只是"问题识别"。事后看，pipeline 对 jsonl 没本地快照是结构盲区，入口应该 `cp -al` 硬链接到 `$RUN_DIR/raw-sessions/`，这样才能让上游硬删变成"历史数据留存"而非"永久丢失"。这条规则用到了 04-24 补跑日报时去引用化的经验：事实锚点（日期、PR号、相对路径）不能删，只有会话级溯源可以删——权限边界反过来就是责任承诺。

新规则落盘前必须先把现有 schema/脚本/hook 的不变量列成表逐项核对，否则规则与实现脱钩。04-24 的 agent-nexus PR #4 把防污染 hook 从补丁层提到根因层（把状态物化到 `superseded/` 目录后 hook 简化为纯路径判断），审查时才发现 `status: superseded` 不在 schema、docs-read 脚本不识别该状态、placeholder 仍在 active 路径——这说明设计前没有不变量清单。类似的模式在 04-23 的 ADR 时也出现了：TDD 铁律却零测试，bootstrap 的 exports dict 与 summary JSON 字段不匹配导致守卫逻辑反而放行。用户在 PR review 时说"这和既有脚本接口怎么衔接"的追问，往往就是在说"你没列不变量表"。一次花 15 分钟列表比事后反复修要快。

高压下跳过验证步骤是系统性问题，不是单点疏忽。04-27 当天三件事共同模式：CC CLI 契约凭记忆写 `--cwd` 直接导致 daemon 启动 abort、GITHUB_TOKEN 透传 sandbox 19 分钟后自己回滚、agent-sandbox 端到端 verify.sh 八条断言未跑。这些都发生在身份迁移（sentixA 被封号）+ agent-nexus 文档治理并行、PR merge 节奏密集的同一个 24 小时。共同特征是"用记忆和直觉代替查证"——CC CLI help fixture 没查就凭记忆写、sandbox 边界凭直觉穿透、脚本凭局部测试通过就略过端到端。当上下文压力大到装不下完整决策树时，大脑会自动做"用已知经验代替查证"的交易。04-19 学过的"追问是测立场"、04-22 学过的"文档建议不等于行动"，在压力下照样失效——这说明规则需要落到机械门或清单里，靠脑力约束会被环境覆盖。

最反直觉的一件事：工程纪律在一个领域做得越完善，越容易在另一个领域产生过度信任。daily-report 的 validator 脚本做到 112 用例全绿，是因为把约束从 prompt 层硬化到脚本层；但同一时期 bootstrap 的 env 跨脚本传递、cc-connect 的数据快照、agent-nexus 的 schema 写法这些问题照样踩坑。一次成功的工程化过程可能给了"我的规范流程是对的"错觉，结果在下一个子系统里又搭起了新的脚手架。真正的成长不是规范越来越多，而是认识到"规范本身也需要被规范"——不是加更多的文档，而是扩大"在动手前先查现有系统的不变量"这个习惯的覆盖面。
